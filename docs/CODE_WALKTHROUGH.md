# Code Walkthrough

A guided tour of the `llm-kb-skills` codebase -- forked from [rvk7895/llm-knowledge-bases](https://github.com/rvk7895/llm-knowledge-bases).

---

## What This Project Is

A Claude Code plugin that turns raw research material into an LLM-maintained Obsidian wiki. Users drop source material into `raw/`, run a skill, and Claude compiles interlinked wiki articles, answers questions, runs health checks, and continuously improves the knowledge base.

The LLM owns the wiki. You explore it in Obsidian; you rarely edit it by hand.

## Repository Layout

```
llm-kb-skills/
  .claude/
    settings.local.json            # Local Claude Code permissions
  .claude-plugin/
    marketplace.json               # Plugin marketplace registration
  plugins/kb/
    .claude-plugin/
      plugin.json                  # Plugin manifest (name, version, repo)
    skills/
      kb-init/SKILL.md             # Bootstrap a new knowledge base
      kb/SKILL.md                  # Main skill: compile, query, lint, evolve
      research/SKILL.md            # Generate a research outline
      research/validate_json.py    # Validate research JSON against field defs
      research-deep/SKILL.md       # Parallel deep-research agents
      research-add-fields/SKILL.md # Add fields to a research outline
      research-add-items/SKILL.md  # Add items to a research outline
      research-report/SKILL.md     # Compile research results into a report
    setup.sh                       # Automated Obsidian plugin installer
  docs/
    plans/                         # Historical design & implementation plans
  CLAUDE.md                        # Project instructions for Claude
  README.md                        # Public documentation
```

After a user runs `/kb-init`, their project gets additional directories:

```
raw/           # Source documents (articles, papers, transcripts, etc.)
wiki/          # LLM-compiled Obsidian vault (never edit manually)
output/        # Query results, reports, lint output
kb.yaml        # Configuration generated by kb-init
```

## Plugin Registration

Two JSON files register this project with the Claude Code plugin system.

### `.claude-plugin/marketplace.json`

Top-level marketplace entry. Declares the package name (`llm-kb-skills`), owner (`renlon`), and lists one plugin: `kb` at `./plugins/kb`.

### `plugins/kb/.claude-plugin/plugin.json`

The plugin manifest. Contains the version (1.3.0), description, license (MIT), and links back to the GitHub repository.

## Skills

Each skill is a `SKILL.md` file with YAML frontmatter (name, description, triggers) followed by a detailed prompt that Claude follows when the skill is invoked. Skills are the core of the project -- they are essentially structured instructions that tell Claude exactly how to perform each operation.

### kb-init -- Bootstrap Setup

**File:** `plugins/kb/skills/kb-init/SKILL.md`

One-time (or re-runnable) setup. Walks the user through a series of choices:

1. Create a new vault or use an existing directory?
2. How to gather raw data? (self-serve, assisted, or mixed)
3. Which output formats? (markdown, Marp slides, matplotlib charts, HTML, CSV, Excalidraw)
4. Maintenance cadence? (`/loop`, scheduled, or manual)

Then it:

- Generates `kb.yaml` with all configured paths and preferences
- Scaffolds `raw/` subdirectories (`articles/`, `papers/`, `repos/`, `notes/`, `images/`, `transcripts/`, `datasets/`)
- Creates `wiki/` and `output/`
- Writes `CLAUDE.md` and `README.md` for the new project
- Runs `setup.sh` to install Obsidian plugins

### kb -- Main Operating Skill

**File:** `plugins/kb/skills/kb/SKILL.md`

The workhorse. Four workflows behind one skill:

#### Compile

Triggered when the user adds files to `raw/` and says "compile" or "update wiki".

1. Reads `wiki/_index.md` to find previously compiled sources and their hashes
2. Scans `raw/` to detect new, changed, and deleted files
3. Dispatches **sonnet** subagents to extract concepts from each new/changed source
4. Creates or updates wiki articles in Obsidian format (YAML frontmatter, `[[wikilinks]]`, source attribution)
5. Updates four index files: `_index.md`, `_sources.md`, `_categories.md`, `_evolution.md`

Key design decision: compilation is **incremental**. It never reprocesses the entire `raw/` folder -- only what changed.

#### Query

Triggered when the user asks a question about the knowledge base.

Three depth levels:

| Depth | What it does |
|-------|-------------|
| **Quick** | Reads wiki indexes and summaries only. No external calls. |
| **Standard** | Cross-references the full wiki, supplements gaps with web search. |
| **Deep** | Full research pipeline: generates an outline (`/research`), launches parallel agents (`/research-deep`), opus synthesizes. |

After every query, a **background auto-evolve subagent** silently checks whether the response contained new knowledge. If it did, the subagent files it into the wiki (new articles, enriched existing ones, updated indexes). If not, it exits silently. The user is never prompted for this.

#### Lint

Health checks for the wiki. Runs parallel **sonnet** subagents to detect:

- Broken `[[wikilinks]]`
- Orphan articles (zero inbound links)
- Uncompiled sources in `raw/`
- Stale articles (source changed since last compile)
- Conflicting claims across articles
- Missing bidirectional backlinks
- Sparse articles (under ~200 words)

Output goes to `output/lint-YYYY-MM-DD.md`, grouped by severity: critical > warning > info.

#### Evolve

Proactive improvement suggestions. Opus reads the wiki structure and dispatches sonnet subagents to find:

- Gaps (concepts referenced but missing articles)
- Cross-category connections
- Claims worth verifying or enriching
- Interesting unanswered questions

Suggestions are ranked and presented to the user, who picks which ones to execute.

### Research Pipeline (4 skills)

These skills were adapted from [Weizhena/Deep-Research-skills](https://github.com/Weizhena/Deep-Research-skills) and translated from Chinese to English.

#### research -- Outline Generation

**File:** `plugins/kb/skills/research/SKILL.md`

`/research <topic>` generates two YAML files in a `{topic_slug}/` directory:

- **`outline.yaml`** -- List of items to research, plus execution config (batch_size, items_per_agent, output_dir)
- **`fields.yaml`** -- Field definitions with categories and detail levels (Minimal/Brief/Detailed)

Uses the model's internal knowledge first, then supplements with one background web-search agent.

#### research-deep -- Parallel Execution

**File:** `plugins/kb/skills/research-deep/SKILL.md`

`/research-deep` reads the outline, launches parallel **sonnet** background agents (one per batch of items), and each agent performs web searches to fill in every field. Results are saved as JSON files in the output directory. Supports resume from checkpoint -- skips items that already have completed JSON files.

Each result JSON is validated against `fields.yaml` using `validate_json.py`.

#### research-add-fields / research-add-items

**Files:** `plugins/kb/skills/research-add-fields/SKILL.md`, `plugins/kb/skills/research-add-items/SKILL.md`

Utility skills to extend an existing research outline. `research-add-fields` appends new field definitions to `fields.yaml`; `research-add-items` appends new research targets to `outline.yaml`. Both deduplicate against existing entries.

#### research-report -- Summary Generation

**File:** `plugins/kb/skills/research-report/SKILL.md`

`/research-report` reads all completed JSON results, generates a Python script (`generate_report.py`), and runs it to produce a markdown report with:

- A table of contents with anchor links and user-chosen summary fields
- Detailed sections organized by field category
- Handling for nested JSON, uncertain values, and fields not in `fields.yaml`

## Model Strategy

The project is intentional about which model handles which task:

| Task | Model | Why |
|------|-------|-----|
| Orchestration, synthesis, final decisions | **Opus** | Highest reasoning quality |
| Wiki article writing, research agents, lint detection | **Sonnet** | Good quality at lower cost |
| Index scanning, link checking, file diffing | **Haiku** | Fast and cheap for mechanical work |

Skills always set the `model` parameter explicitly when dispatching subagents.

## Setup Script

**File:** `plugins/kb/setup.sh`

A bash script that automates Obsidian community plugin installation. Runs during `/kb-init`.

**Required plugins:**
- **Dataview** -- Enables SQL-like queries over wiki frontmatter
- **Obsidian Git** -- Automatic backups via git

**Optional plugins:**
- Kanban, Outliner, Tag Wrangler, Local Images Plus

Three modes: interactive (prompts for each optional plugin), `--all` (install everything), or `--only id1,id2` (specific plugins).

The script downloads plugin releases from GitHub, writes config to `.obsidian/`, and provides instructions for the Obsidian Web Clipper browser extension.

## Configuration

### kb.yaml (generated by kb-init)

Central config file read at the start of every skill invocation. Contains:

- **paths** -- Locations of `raw/`, `wiki/`, `output/`
- **output_formats** -- Which formats to support (markdown, marp, matplotlib, etc.)
- **obsidian** -- Vault path and plugin settings
- **integrations** -- Optional tool configs (e.g., Smaug path for X/Twitter)

### CLAUDE.md

Project instructions loaded into every Claude session. Establishes:

- The wiki is the LLM's domain -- never manually edit it
- Always read `kb.yaml` before operating
- Obsidian formatting rules (`[[wikilinks]]`, YAML frontmatter)
- Model dispatch strategy

### .claude/settings.local.json

Local permissions: which skills can be invoked, which bash commands are allowed (git, python3, pip), and whether web search is enabled.

## X/Twitter Integration

An optional feature in the `kb` skill. If [Smaug](https://github.com/alexknowshtml/smaug) is installed and its path is configured in `kb.yaml`, the skill can fetch tweet content via `npx smaug fetch <tweet_id>`. Without Smaug, users can still paste tweet text manually or use Thread Reader App for threads. The skill never attempts `WebFetch` on X URLs (they're auth-walled).

## Key Design Decisions

1. **Obsidian-native output** -- Everything uses `[[wikilinks]]` and YAML frontmatter so the wiki works directly in Obsidian without conversion.

2. **Incremental compilation** -- Only new/changed sources are processed. The `_index.md` file tracks source hashes to enable diffing.

3. **Index-first querying** -- Questions are answered by reading wiki indexes first, avoiding the need for vector search or RAG infrastructure.

4. **Silent auto-evolve** -- After every query, a background subagent checks for new knowledge and files it into the wiki without user intervention.

5. **Cost-aware model dispatch** -- Cheap models for mechanical tasks, expensive models for reasoning. Every subagent call specifies its model explicitly.

6. **Graceful degradation** -- Optional integrations (Smaug, output formats) degrade gracefully when unavailable instead of failing.
