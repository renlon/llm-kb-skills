# Episode Index & Dedup Design

**Status:** Design in progress (2026-04-21)
**Skills:** `plugins/kb/skills/kb-publish/` (writer) + `plugins/kb/skills/kb-notebooklm/` (reader)
**Related:** Built on the podcast intro-music + transcript feature (spec 2026-04-20), which ships speaker-labeled transcripts for every new episode.

## Problem

The podcast workflow's current "published-topic cross-check" (`kb-notebooklm` step 5b) does a simple string match against `episodes.yaml`'s `concepts_covered[]` — itself populated from source-lesson headings, not from what actually aired. This creates three failure modes:

1. **Paraphrased concepts slip through.** "FlashAttention" vs "Flash Attention" — different strings, same concept.
2. **The index lies about depth.** A lesson heading says "KV Cache" but the episode only name-dropped it; the registry says "covered" when it was really "mentioned."
3. **No way to detect "deeper-dive" opportunities.** Even if overlap is detected, we can't tell the user "EP2 covered this at intro level; frame the new episode as a deep-dive follow-up."

The deeper issue: we don't have a structured record of what the audience has actually been taught. Without that, every new episode risks repeating material or — worse — repeating material with no awareness that we're doing it.

## Goals

- Produce a structured, queryable record of every published episode's concepts-covered + key-points-taught, derived from the actual transcript.
- Canonicalize concepts against the existing wiki — "Flash Attention", "flash-attention", "FlashAttention" all resolve to `[[wiki/attention/flash-attention]]`.
- Replace the string-match cross-check with a depth-aware dedup judge that distinguishes "novel", "deeper_dive", "addresses_open_thread", "redundant".
- Keep the user in the loop: the user-facing confirmation prompt in step 5b shows structured recommendations, not raw data.
- Backfill the two already-published episodes (EP1, EP2) into the new index.

## Non-Goals (YAGNI)

- No SQLite, no FTS5, no embeddings, no vector DB. The wiki itself is the data store. (An earlier draft of this spec proposed SQLite; superseded.)
- No graph-database software (Neo4j etc). Obsidian's wikilink format IS the graph; Python parses it directly.
- No automatic wiki-article rewriting for existing concepts. Auto-stubs only create new articles; they never overwrite existing human-written content.
- No cross-language concept deduplication beyond what the Haiku canonicalization does at extract time.
- No changes to the podcast generation or transcription pipeline (that's the spec 2026-04-20 feature, already shipped).

## Design Overview

Two layers, both backed by the Obsidian vault (which is already the KB's primary data store):

- **Layer 1 (at publish time):** `kb-publish` reads the episode's transcript, prompts Haiku to extract structured concepts + key_points, canonicalizes them as wikilinks, auto-stubs any missing concept articles, and writes an `wiki/episodes/ep-<id>-<slug>.md` index record.
- **Layer 3 (at podcast-generation time):** `kb-notebooklm` reads all `wiki/episodes/*.md`, extracts candidate concepts from the proposed new episode's source lessons, asks Haiku to classify each candidate's relationship to past episodes (novel / deeper_dive / addresses_open_thread / redundant), and presents structured recommendations to the user.

There is no Layer 2. The earlier draft's "SQLite FTS5" layer is unnecessary — the wiki articles are already structured data, and Obsidian itself provides search + graph visualization for free.

## Detailed Design

### 1. Episode wiki article format

One markdown file per published episode at `wiki/episodes/ep-<id>-<slug>.md`:

```markdown
---
title: "EP3 | 量化：让大模型装进你的显卡"
episode_id: 3
audio_file: podcast-quantization-2026-04-21.mp3
transcript_file: podcast-quantization-2026-04-21.transcript.md
date: 2026-04-21
depth: deep-dive
tags: [episode, quantization, deployment]
aliases: ["EP3", "量化episode"]
source_lessons:
  - Model_Quantization_and_Formats_2026-04-07.md
  - k-quants-model-quantization_2026-04-07.md
  - Model_File_Formats_2026-04-07.md
---

# EP3 | 量化：让大模型装进你的显卡

**Date:** 2026-04-21
**Depth:** deep-dive
**Audio:** [[podcast-quantization-2026-04-21.mp3]]
**Transcript:** [[podcast-quantization-2026-04-21.transcript]]

## Summary

{1-2 paragraph summary generated by Haiku from the transcript}

## Series Links

- Builds on: [[wiki/episodes/ep-01-gpu-computing-cuda]] (hardware context)
- Follow-up candidates: inference serving frameworks deep-dive; speculative decoding
```

**Rationale:**

- **Wikilinks are the primary dedup signal.** The Layer 3 judge only needs to ask: "does any episode's `## Concepts Covered` section wikilink to the same slug as a candidate concept in a new proposal?"
- **Depth is encoded structurally** via the `### Deep-dive / Explained / Mentioned` sub-headings, visible in Obsidian.
- **Key points are specific teaching claims**, not generic summaries. This is what distinguishes "genuinely redundant" from "same topic, different angle" at query time.
- **Series Links is optional** but useful — Obsidian's Graph view visualizes episode-to-episode connections for free.
- **The existing `episodes.yaml` registry stays** as the system-of-record for publish state (`generated → draft → published`). The wiki article is the knowledge record. They serve different purposes.

**Parser strategy — structured frontmatter block + rendered markdown:**

The visible markdown is for Obsidian's graph/graph-view/human-reading. The authoritative structured data lives in a machine-readable YAML block in the frontmatter, generated alongside the markdown and parsed back deterministically. The parser reads this block; it never regex-parses the rendered markdown.

Full frontmatter schema:

```yaml
---
# Human-visible metadata (existing conventions)
title: "EP3 | 量化：让大模型装进你的显卡"
episode_id: 3
audio_file: podcast-quantization-2026-04-21.mp3
transcript_file: podcast-quantization-2026-04-21.transcript.md
date: 2026-04-21
depth: deep-dive
tags: [episode, quantization, deployment]
aliases: ["EP3", "量化episode"]
source_lessons:
  - Model_Quantization_and_Formats_2026-04-07.md
  - k-quants-model-quantization_2026-04-07.md
  - Model_File_Formats_2026-04-07.md

# Machine-readable structured data (THIS is what the parser reads)
index:
  schema_version: 1
  summary: "..."
  concepts:
    - slug: wiki/quantization/k-quants
      depth_this_episode: deep-dive
      depth_delta_vs_past: new
      prior_episode_ref: null
      what: "Group-wise quantization with per-group scale/zero-point."
      why_it_matters: "Enables 4-bit inference with minimal quality loss."
      key_points:
        - "Groups tensor values into blocks of ~32-256 elements"
        - "Per-group scale avoids range-collapse on outliers"
      covered_at_sec: 252.3
      existed_before: true
  open_threads:
    - slug: wiki/quantization/qat
      existed_before: true
      note: "Quantization-Aware Training"
  series_links:
    builds_on: [wiki/episodes/ep-01-gpu-computing-cuda]
    followup_candidates: ["inference serving frameworks deep-dive"]
---
```

The body markdown below the frontmatter is rendered FROM the `index` block for humans to read. If a user hand-edits the body, the edits are preserved until the next reindex (which rewrites the body from the structured data). If a user hand-edits the `index` block itself, the next parse picks up the edit — this is a feature: the user can manually fix Haiku mistakes by editing YAML.

This makes `scan_episode_wiki()` a thin wrapper around `yaml.safe_load` — no regex, no markdown parsing, no brittleness.

### 2. Layer 1 — Concept Extraction (at publish time)

**Where it runs:** New step 8c in `kb-publish`, after a successful publish AND after the sidecar has been imported. Only when `transcript_applied: true` for the episode. Also runs during backfill (see section 5).

**Why at publish time, not generation time:** The transcript may be touched up by the user between generation and publish (e.g., correcting diarization errors). Indexing at publish time captures the final audience-facing content.

**Inputs:**
- Episode transcript markdown file.
- Episode metadata from `episodes.yaml`: `id`, `title`, `date`, `depth`, `topic`, `source_lessons`.
- **Concept catalog**: list of existing wiki articles (slugs + titles + tags) from `<vault>/wiki/` via `concept_catalog(wiki_dir, include_stubs=True)`. `wiki/episodes/**` is always excluded. Stubs ARE included (marked `is_stub: true`) so Haiku canonicalizes to them and doesn't create duplicates. Grouped by top-level category to bound input-token size.
- **Recent-episodes context**: for the 3 most recent published episodes plus any episodes whose frontmatter `tags` overlap with the current episode's `tags`, include `{ep_id, title, depth, concepts-covered-slugs, open_threads}`.

**Prompt template:** `plugins/kb/skills/kb-publish/prompts/episode-wiki-extract.md`. Instructs Haiku to produce a single JSON object with:
- `summary` (1-2 paragraph)
- `concepts[]` with fields `slug`, `depth_this_episode`, `what`, `why_it_matters`, `key_points[]`, `covered_at_sec`, `existed_before`
- `open_threads[]` with `{slug, existed_before, note}`
- `series_links` with `builds_on[]` and `followup_candidates[]`

**Slug convention.** Every `slug` field anywhere in this spec is the vault-relative path WITHOUT the `.md` extension, always starting with `wiki/` (e.g., `wiki/quantization/k-quants`). This matches how Obsidian renders the wikilink `[[wiki/quantization/k-quants]]`. When a slug is rendered into a markdown wikilink, we wrap it in `[[ ]]`. When it's written to YAML frontmatter or a JSON response, it's a bare string. Never includes trailing `.md`, never starts with `/`.

**Slug validation (security + sanity):** Before writing to any path derived from a slug:
- Must match `^wiki/[a-z0-9][a-z0-9\-_./]*[a-z0-9]$` — lowercase, hyphens/underscores/dots/slashes allowed in the middle.
- Must NOT contain `..` at any path component.
- Must NOT start with `wiki/episodes/` unless writing an episode article (so Haiku can't propose a "concept" that collides with an episode record).
- Must NOT end with `.md` — the extension is added at write time only.
- Must NOT exceed 180 chars.
- On macOS (case-insensitive filesystem): check for case-only collisions against existing files. If a slug `wiki/quantization/K-Quants` would collide with an existing `wiki/quantization/k-quants`, normalize to the existing casing.
- `wiki_dir` is always the vault-relative root (typically `<vault>/wiki/`). Path assembly is `wiki_dir.parent / slug` — this correctly produces `<vault>/wiki/quantization/k-quants` without double-nesting.

**Depth-delta is NOT asked of Haiku** (it cannot reliably prove "new across all episodes" from partial context). Haiku emits `depth_this_episode` only. Python computes `depth_delta_vs_past` post-extraction:

```
depth_order = {"mentioned": 0, "explained": 1, "deep-dive": 2}
for concept in extracted.concepts:
    prior = coverage_map.get(concept.slug, [])  # all prior episodes that covered this slug
    if not prior:
        concept.depth_delta_vs_past = "new"
        concept.prior_episode_ref = None
    else:
        deepest_prior = max(prior, key=lambda p: depth_order[p.depth])
        dp = depth_order[deepest_prior.depth]
        dc = depth_order[concept.depth_this_episode]
        if dc > dp:
            concept.depth_delta_vs_past = "deeper"
            concept.prior_episode_ref = deepest_prior.ep_id
        elif dc == dp:
            concept.depth_delta_vs_past = "same"
            concept.prior_episode_ref = deepest_prior.ep_id
        else:
            concept.depth_delta_vs_past = "lighter"
            concept.prior_episode_ref = deepest_prior.ep_id
```

`coverage_map` is built from all `wiki/episodes/*.md` via `scan_episode_wiki()` + `concepts_covered_by_episodes()`, **FILTERED to exclude the current episode being indexed** (by `episode_id`). This prevents reindex from self-counting prior coverage: on `--reindex 3`, the existing `wiki/episodes/ep-03-*.md` is skipped while building the coverage_map. If no episode filter were applied, every concept in a reindexed episode would show `depth_delta_vs_past = "same"` with `prior_episode_ref = <self>` — a nonsense signal.

"Prior" for tie-breaking when multiple episodes share the same deepest depth: prefer the episode with the **lowest `episode_id`**. This is deterministic and matches listener chronology (episodes are numbered in publish order).

The Haiku prompt still gets a recent-episodes context block for canonicalization quality (so it canonicalizes to existing slugs), but the depth-delta decision is deterministic Python.

**Validation:**

1. Parse JSON (retry once on malformed output — Haiku occasionally wraps in code fences despite instructions).
2. Verify required keys present, `depth_this_episode` in allowed set `{mentioned, explained, deep-dive}`.
3. Validate every slug against the rules above.
4. **Recompute `existed_before` from the filesystem**, not from Haiku's claim: `existed_before = Path(<vault>/<slug>.md).exists()`. This prevents the model mistaking an existing article as new (which would attempt a stub-overwrite blocked by collision rules, losing coverage) or a truly-new concept as existing (no stub gets created, wikilink resolves to a red link).
5. On any validation failure, abort indexing for this episode with a clear error. Do NOT write partial data.

**Cost:** Haiku 4.5 input ~10-15k tokens, output ~3k. ~$0.02 per episode. Negligible.

**Idempotency (transactional reindex):** Indexing writes to a staging directory first, then atomically moves artifacts into place only after full success.

**Staging layout:** `<vault>/.kb-publish-staging/<session-uuid>/`. Directory structure is **wiki-root-relative** — i.e., what lives inside `<vault>/wiki/`. Slugs always include the `wiki/` prefix but staging paths do NOT (that prefix is stripped once, at staging-path derivation):

```
<vault>/.kb-publish-staging/<session-uuid>/
    episodes/ep-03-quantization.md           # from slug "wiki/episodes/ep-03-quantization"
    quantization/k-quants.md                 # from slug "wiki/quantization/k-quants"
    formats/gguf.md                          # from slug "wiki/formats/gguf"
    formats/safetensors.md                   # from slug "wiki/formats/safetensors"
```

Every staged path is derived from its slug via `slug_to_wiki_relative_path(slug)` (see section 7). At commit time, `os.replace(staging_dir / rel, wiki_dir / rel)` moves each file into the real vault. The `strict=True` smoke parse passes the staging dir as wiki_dir to the scanner, which looks at `<passed-in wiki_dir>/episodes/*.md` — the staging dir matches that convention because episodes are at `staging/episodes/ep-<id>-<slug>.md`.

**Flow:**

1. Create staging dir. Generate episode filename slug (see "Episode filename slug validation" below).
2. Render each auto-stub into its staging path — but only for concepts with `existed_before: false`. For existing stubs colliding with new content, compute the stub-update diff in memory (not written until commit).
3. Render the new episode article into `staging/episodes/ep-<id>-<slug>.md`.
4. Validate (section 2 validation for the episode's JSON; run `scan_episode_wiki(wiki_dir=staging_dir)` with `strict=True` as a parse round-trip smoke test — the scanner looks at `<wiki_dir>/episodes/*.md` relative to whatever `wiki_dir` is passed, so this works unchanged).
5. **Atomic commit phase, in order:**
   a. For each staged stub at `staging/<rel>`: check the corresponding destination `<vault>/wiki/<rel>`. If a non-stub article is already there, skip this stub (existing article is canonical). Otherwise `os.replace(staging/<rel>, <vault>/wiki/<rel>)`.
   b. For each existing stub-update diff (computed in step 2): update frontmatter fields in-place via read-modify-write with `os.replace()`.
   c. **Finally**, `os.replace(staging/episodes/ep-<id>-<slug>.md, <vault>/wiki/episodes/ep-<id>-<slug>.md)`. The episode article lands LAST so its wikilinks always resolve at the moment it becomes visible.
6. Clean up staging dir.

If any step 1–4 fails: abort, leave the existing index untouched. The prior valid index record is preserved.

If step 5 fails partway through (rare — os.replace on same filesystem is atomic but multi-file isn't): log the partially-committed state, surface to the user for manual check, but don't attempt rollback (that would require keeping the pre-replace content around, adding complexity for a rare case). In practice `os.replace` on a local filesystem is reliable enough that this partial-commit path is unlikely.

The old `delete-first-then-write` approach is explicitly rejected; `--reindex <id>` is safe via replacement, not pre-deletion.

**Episode filename slug validation:** The episode filename slug (the `<slug>` in `ep-<id>-<slug>.md`) is derived from the episode's `topic` field in `episodes.yaml`, normalized via this deterministic algorithm:

1. If `episode.topic` contains any ASCII letters, use topic as input. Otherwise, use the first explicit `English subtitle / latin version` alias from `episode.aliases`, if any. Otherwise, fall through.
2. Normalize (ASCII-only):
   - Lowercase.
   - Replace any run of non-`[a-z0-9]` chars with a single hyphen.
   - Strip leading/trailing hyphens.
   - Truncate to 50 chars.
3. If the normalized result is non-empty AND matches `^[a-z0-9][a-z0-9\-]*[a-z0-9]$`: use it as the slug.
4. Otherwise (pure non-ASCII topic with no Latin alias, normalized to empty, rules violated): **fallback slug = `topic`** where `topic` is the string `"topic"` — i.e., the filename becomes `ep-<id>-topic.md`. This is intentionally plain; the user can manually rename the file and update `episodes.yaml` if they want a richer slug. The guaranteed-deterministic fallback ensures indexing never fails on Chinese-only topics.

No transliteration dependency — Chinese `"模型量化 & 部署"` with no Latin alias normalizes to empty, falls back to `ep-<id>-topic.md`.

Same slug-safety rules as concept slugs: no `..`, no absolute paths, final slug must match `^[a-z0-9][a-z0-9\-]*[a-z0-9]$`.

On macOS (case-insensitive filesystem): check for case-insensitive collision against existing `wiki/episodes/*.md`:
- Collision on the SAME episode_id (prefix match `ep-<id>-`): expected during reindex; existing file gets replaced via `os.replace()`.
- Collision on a DIFFERENT episode_id: error (should not happen with valid `episodes.yaml`; surface to user).

### 3. Auto-stub for new concepts

When Haiku returns a concept with `existed_before: false`, write a stub article at the proposed slug. This keeps the wikilink graph connected ("green") — Obsidian's default red-for-unresolved wikilinks gets noisy fast otherwise.

**Stub format** (written at `<vault>/<slug>.md`, i.e. the slug already contains the `wiki/` prefix):

```markdown
---
title: "{Concept Name}"
tags: [stub, {inferred-tag}]
status: stub
created_by: ep-{N}              # immutable — the episode that introduced this stub
last_seen_by: ep-{M}            # mutable — most recent episode to reference this concept
best_depth_episode: ep-{K}      # mutable — episode that covered it at greatest depth
best_depth: explained           # mutable — deepest depth any episode has reached
referenced_by: [ep-1, ep-3]     # mutable — every episode that has wikilinked to this
created: {iso-date}
aliases: []
---

# {Concept Name}

> **Stub.** Auto-created by `kb-publish` while indexing [[wiki/episodes/ep-{N}-{slug}]]. It will be fleshed out the next time this concept appears in compile/lint, or when you write about it manually.

## What the introducing episode said

{concept.what from the Haiku extraction of the `created_by` episode}

## Why it matters

{concept.why_it_matters}

## Referenced by

- Introduced in: [[wiki/episodes/ep-{created_by}-...]]
- Latest reference: [[wiki/episodes/ep-{last_seen_by}-...]]
- Best coverage: [[wiki/episodes/ep-{best_depth_episode}-...]] ({best_depth})
```

**Collision handling:**

- If a slug Haiku proposes already has a non-stub article (i.e., `status != stub`): DO NOT overwrite. The existing article is canonical. Log and proceed. Do update the episode article's wikilink to point at the canonical article.
- If a slug has an existing stub from a different episode: treat `created_by` as immutable provenance. Update only `last_seen_by`, `best_depth_episode` (if this episode covered it deeper), `best_depth`, and append to `referenced_by`. Leave all prose sections untouched.
- If a slug has an existing stub created by the same episode (reindex case): let the transactional-reindex flow fully replace the stub with a freshly-rendered one.
- Stubs are idempotent — re-running auto-stub produces no new writes when state hasn't changed.

**Stubs in the canonicalization catalog:** Stubs ARE included in the catalog passed to Haiku, tagged with a marker `(stub — weak canonical record)`. This lets Haiku canonicalize to stubs (preventing duplicate stubs), while signaling that the content is provisional. The `concept_catalog()` helper supports both modes via an `include_stubs: bool = True` parameter; default is include.

**What stubs do NOT do:**
- They don't attempt a full wiki article. That's the kb compile/evolve workflow's job.
- They don't generate diagrams or deep explanations.
- They don't touch articles outside `wiki/` (e.g., they never write to `lessons/` or `output/`).

### 4. Layer 3 — Dedup Judge (at podcast-generation time)

**Where it runs:** `kb-notebooklm` podcast workflow step 5b, replacing the existing string-match "Published topic cross-check."

**Inputs:**
- Proposed episode grouping from step 5: theme name + source lesson list.
- Parsed `wiki/episodes/*.md` via `episode_wiki.scan_episode_wiki(wiki_dir)`.
- Rough candidate concepts extracted from each source lesson's title + first 400 tokens (cheap pre-pass).

**Confidentiality filter compliance:** Pre-compute MUST read source material from the SAME files that step 6g of the podcast workflow will eventually upload — i.e., the sanitized variants (from `config.output_path/sanitized/*.sanitized.md`) for any lesson that the step 4b filter flagged as `sanitizable`, the raw lesson files for `generic` lessons, and skipped lessons excluded entirely. Under no circumstances does the dedup pre-pass read a raw internal-tagged lesson. This means step 5b executes AFTER step 4b — the execution order in `kb-notebooklm/SKILL.md` must be amended to reflect this, moving step 5b to fall after 4b's classification + sanitization.

**Pre-compute** (Python, no LLM):
1. `all_episodes = scan_episode_wiki(wiki_dir)`
2. `coverage = concepts_covered_by_episodes(all_episodes)` — dict of `{slug: [{ep_id, depth, key_points}, ...]}`
3. For each candidate concept name, best-effort resolve to a wiki slug via `resolve_concept_candidate()`: exact case-insensitive title match (unique wins), then alias match (unique wins). No tag-based resolution — tags are too broad. Unresolved candidates pass through as raw names (Haiku will canonicalize at judgment time with full context).
4. For each resolved candidate, look up `coverage[slug]` → prior_hits.
5. Collect all `open_threads` from recent episodes.

**Haiku judge call** — one call per proposed episode group (not per concept), using prompt `plugins/kb/skills/kb-notebooklm/prompts/dedup-judge.md`. Haiku is given the candidate list + prior-hits + open-threads, and returns:

```json
{
  "per_concept": [
    {"candidate": "k-quants", "verdict": "redundant_same_depth",
     "reasoning": "EP3 covered k-quants at deep-dive; new proposal would repeat without adding angle or depth.",
     "recommended_framing": "Skip this concept. If the new episode needs to mention it, reference EP3 in one sentence and move on."},
    {"candidate": "compute capability", "verdict": "deeper_dive",
     "reasoning": "EP1 introduced GPU compute at intro depth; this proposal goes concrete about CC numbers.",
     "recommended_framing": "Open with a callback to EP1, then go into CC specifics."}
  ],
  "episode_verdict": "proceed",
  "framing_recommendation": "If proceeding, what angle/depth makes this distinct"
}
```

**Verdict values:**
- `novel` — no prior coverage.
- `deeper_dive` — prior episode covered at shallower depth; adding value.
- `addresses_open_thread` — answers a question hinted at in a prior episode's `open_threads`.
- `redundant_same_depth` — prior episode covered at same depth.
- `redundant_key_points` — prior episode already made the specific teaching claims.

**User-facing output:** the skill renders the judge's response into the same confirmation prompt shape step 5b uses today, just with richer content:

```
Episode group "Model Quantization" — cross-check:

✓ Novel concepts (5):
  - k-quants
  - GGUF format
  - safetensors
  - AWQ (mentioned only — will be fine)
  - GPTQ (mentioned only)

⚠ Deeper-dive opportunities (1):
  - Compute Capability
    Prior: EP1 covered GPU fundamentals at intro depth.
    Recommended framing: "Now let's get concrete about what GPUs can actually run."

⛔ Potential redundancies: (none)

Episode verdict: proceed (as deep-dive)
Framing: Position as the technical companion to EP1 — hardware constraints
are the 'why' behind quantization's 'what'.

Proceed? (y / adjust / skip)
```

The user either approves, asks for changes to the grouping, or skips.

### 5. Integration — kb-publish new step 8c

After step 8b's registry update succeeds:

```
8c. Episode wiki indexing (best-effort, non-publish-blocking, transactional)

This step calls `orchestrate_episode_index()` for the full extraction-and-indexing pipeline; that helper delegates filesystem writes to `index_episode_transactional()`. Backfill uses the same `orchestrate_episode_index()` helper.

1. Read the current episode's registry entry. The fields we need:
   - `audio` (basename of the distributed mp3 — already exists in the schema)
   - `transcript.applied` (nested bool, set during publish)
   - `transcript.markdown` (nested basename of the transcript file, relative to the audio's parent dir)

   If `transcript.applied` is false or `transcript.markdown` is null:
     log "Skipping episode indexing — no transcript available."
     return (episode still published successfully)

2. Read the episode's `.transcript.md`.
3. Build concept catalog via `concept_catalog(wiki_dir, include_stubs=True)`. Stubs are included so Haiku canonicalizes to them.
4. Build recent-episodes context from `wiki/episodes/*.md` via `scan_episode_wiki(wiki_dir, strict=False)` filtered to the 3 most-recent + any whose `tags` overlap with this episode's `tags`.
5. Call Haiku with the episode-wiki-extract prompt.
6. Validate the JSON response (section 2 validation). On failure, log and return — no wiki writes, no registry update.
7. Compute depth_deltas via `compute_depth_deltas(concepts, coverage_map)` from all-episodes coverage.
8. Call `index_episode_transactional(...)`:
   - Creates staging dir.
   - Stages the rendered episode article at `staging/episodes/ep-<id>-<slug>.md`.
   - For each `existed_before: false` concept: stages a freshly-rendered stub at `staging/<concept-slug>.md`.
   - For each `existed_before: true` concept that IS currently a stub:
     - If the existing stub's `created_by == f"ep-{this episode_id}"` (same-episode reindex): stage a FULL replacement stub (prose + frontmatter).
     - Otherwise (different episode's stub): stage a frontmatter-only update record (in-memory; applied at commit time via read-modify-write that preserves prose sections).
   - Runs `scan_episode_wiki(staging_dir, strict=True)` as smoke-parse over the staging episode article. On failure, aborts and cleans up staging.
   - Commits atomically (section 2's commit flow): stubs first (skipping non-stub collisions), then stub-updates (read-modify-write existing stub files to update `last_seen_by`, `best_depth_episode`, `best_depth`, `referenced_by` — never `created_by`), finally the episode article via `os.replace()`.
   - Cleans up staging.
   - Returns a `TransactionalIndexResult` dataclass with `{new_stubs_created: [slugs], stubs_updated: [slugs], collisions_skipped: [slugs], episode_article: Path}`.
9. Update the episode's entry in `episodes.yaml` (separate transaction, after wiki commit):
   - Replace `concepts_covered[]` with a list derived from the extraction (one entry per concept with `{name, depth}` — the full key_points stay in the wiki article).
   - Replace `open_threads[]` with the extraction's open_threads slugs (stripped of wiki path prefix).
   - DO NOT touch `id`, `title`, `description`, `date`, `status`, `hosts`, `transcript`, `intro_applied`, or `source_lessons`. Those are owned by the publish pipeline.
10. Log success: "Indexed EP<N> with <concept count> concepts; <new_stubs_created> new stubs; <stubs_updated> stubs updated; <collisions_skipped> collisions skipped."

On any failure at steps 2–8: log the error, leave existing wiki/registry unchanged, DO NOT abort the publish. The episode remains published; indexing is retriable via `backfill-index --episode <N>`.

**Partial-failure case — wiki commit succeeded but registry update (step 9) failed:** The wiki index is newer than the registry. This is detectable on next read (wiki episode's `index.concepts[].slug` will contain entries not in `episodes.yaml`'s `concepts_covered[]`). The next backfill-index or reindex call will update the registry. Log clearly for the user. Do NOT attempt to roll back the wiki commit (that would require re-running the full staging process in reverse).
```

### 7. Shared Python module

**File:** `plugins/kb/skills/kb-publish/scripts/episode_wiki.py`
**Tests:** `plugins/kb/skills/kb-publish/scripts/tests/test_episode_wiki.py`

**Public API:**

```python
from dataclasses import dataclass
from pathlib import Path

@dataclass
class IndexedConcept:
    slug: str                        # e.g. "wiki/quantization/k-quants"
    depth_this_episode: str          # mentioned | explained | deep-dive
    depth_delta_vs_past: str         # new | deeper | same | lighter
    prior_episode_ref: int | None
    what: str
    why_it_matters: str
    key_points: list[str]
    covered_at_sec: float | None

@dataclass
class OpenThread:
    slug: str | None         # wiki slug if one exists/was created; None for unresolved threads
    note: str                # human-readable description
    existed_before: bool     # True if slug already has a wiki article; False if auto-stubbed; irrelevant if slug is None

@dataclass
class IndexedEpisode:
    episode_id: int
    title: str
    date: str
    depth: str                       # intro | intermediate | deep-dive
    audio_file: str
    transcript_file: str | None
    concepts: list[IndexedConcept]
    open_threads: list[OpenThread]
    series_builds_on: list[str]
    series_followup_candidates: list[str]

def scan_episode_wiki(
    wiki_dir: Path,
    strict: bool = False,
) -> list[IndexedEpisode]:
    """Parse all wiki/episodes/*.md files into structured records, sorted by episode_id.

    strict=False (default, used by dedup at podcast-generation time): malformed
    frontmatter or missing `index` block logs a warning and skips the file.
    Dedup proceeds with partial coverage; better to be lenient than block
    generation because an unrelated episode file got a bad edit.

    strict=True (used by reindex smoke test): any parse failure raises
    EpisodeParseError. This is what catches a freshly-rendered staging file
    having a bug."""

def concepts_covered_by_episodes(
    episodes: list[IndexedEpisode],
) -> dict[str, list[dict]]:
    """{slug: [{ep_id, depth, key_points, date}, ...]} — what each concept has been taught."""

def concept_catalog(
    wiki_dir: Path,
    include_stubs: bool = True,
) -> dict[str, list[dict]]:
    """{top-level-category: [{slug, title, tags, aliases, is_stub}, ...]} — for Haiku prompts.
    Episode articles (wiki/episodes/**) are always excluded.
    Stubs are included by default (flagged via is_stub=True), so Haiku can
    canonicalize to them and avoid duplicate-stub proliferation."""

def compute_depth_deltas(
    concepts: list[dict],
    coverage_map: dict[str, list[dict]],
) -> list[dict]:
    """Given raw extracted concepts (no depth_delta) and a coverage_map built
    from all prior episodes, return concepts with depth_delta_vs_past and
    prior_episode_ref filled in. Uses the depth_order algorithm from section 2."""

def validate_slug(slug: str, *, allow_episode: bool = False) -> None:
    """Raise SlugValidationError if slug fails any of the section-2 rules.

    allow_episode=False (default): reject slugs starting with 'wiki/episodes/'
    (concept slugs must never collide with episode records).
    allow_episode=True: accept episode article slugs; used only by the episode
    article staging path logic in index_episode_transactional()."""

def staging_dir(wiki_dir: Path) -> Path:
    """Return a fresh ephemeral staging dir for transactional reindex."""

def atomic_replace_index(
    staging: Path, wiki_dir: Path, episode_article_filename: str,
) -> None:
    """Move all files from staging into wiki_dir (episode article last,
    skipping stubs that have been filled in with non-stub content since
    staging was populated). os.replace() for atomicity."""

def render_stub(
    slug: str,
    concept: IndexedConcept,
    episode_id: int,
    episode_slug: str,
    date: str,
) -> str:
    """Deterministic rendering of a stub article (frontmatter + body) from
    structured inputs. Pure function — no filesystem writes."""

def compute_stub_update(
    existing_stub_frontmatter: dict,
    concept: IndexedConcept,
    episode_id: int,
) -> dict | None:
    """Given an existing stub's frontmatter and a concept from a new episode,
    compute the frontmatter update (or None if no update needed).

    Always updates `last_seen_by`.
    Updates `best_depth_episode` + `best_depth` iff the new depth > stored best.
    Appends `referenced_by` if this episode isn't already in the list.
    NEVER modifies `created_by` (immutable provenance).
    Returns the full updated frontmatter dict on change, None on no-op."""

@dataclass
class TransactionalIndexResult:
    episode_article: Path
    new_stubs_created: list[str]       # slugs
    stubs_updated: list[str]           # slugs with frontmatter-only updates
    collisions_skipped: list[str]      # slugs where non-stub articles already exist

def index_episode_transactional(
    wiki_dir: Path,
    episode_id: int,
    episode_topic: str,
    episode_date: str,
    episode_depth: str,
    audio_file: str,
    transcript_file: str,
    tags: list[str],
    aliases: list[str],
    source_lessons: list[str],
    extraction: dict,        # parsed + validated Haiku JSON + depth_deltas applied + existed_before recomputed
) -> TransactionalIndexResult:
    """Filesystem-only transactional indexing: stage, validate via strict-mode
    round-trip parse, commit atomically.

    Caller is responsible for producing `extraction` (catalog/context build,
    Haiku call, validation, existed_before recompute, depth-delta computation).
    See `orchestrate_episode_index()` for the full pipeline.

    Raises TransactionAbortedError if validation or staging smoke-parse fails
    (leaving existing wiki untouched). Returns structured result on success
    for progress logging.

    Same-episode stub handling: for any stub whose `created_by == f"ep-{episode_id}"`,
    stage a FULL replacement (not a frontmatter-only update) — this is the
    reindex-of-introducing-episode case where prose should reflect the latest
    extraction. For stubs created by other episodes, stage a frontmatter-only
    update."""

def orchestrate_episode_index(
    wiki_dir: Path,
    episode_id: int,
    episode_topic: str,
    episode_date: str,
    episode_depth: str,
    audio_file: str,
    transcript_path: Path,
    transcript_file: str,
    tags: list[str],
    aliases: list[str],
    source_lessons: list[str],
    haiku_call: Callable[[str], str],     # injected for testability
) -> TransactionalIndexResult:
    """Full pipeline: read transcript, build catalog+context, call Haiku,
    validate, recompute existed_before, compute depth_deltas excluding current
    episode, invoke index_episode_transactional, return result.

    Used by both step 8c (at publish time) and /kb-publish backfill-index.

    coverage_map is built from `scan_episode_wiki(wiki_dir, strict=False)`
    with the current episode_id filtered out — this prevents reindex from
    self-counting prior coverage."""

def slug_to_wiki_relative_path(slug: str) -> Path:
    """Convert a slug like 'wiki/quantization/k-quants' to the wiki-root-relative
    path 'quantization/k-quants.md'. Strips exactly one leading 'wiki/' prefix
    and appends '.md'. Used by staging and commit to avoid 'wiki/wiki/...'
    double-nesting."""

def render_episode_wiki(
    episode_id: int,
    title: str,
    date: str,
    depth: str,
    audio_file: str,
    transcript_file: str,
    summary: str,
    concepts: list[IndexedConcept],
    open_threads: list[dict],
    series_builds_on: list[str],
    series_followup_candidates: list[str],
    source_lessons: list[str],
    tags: list[str],
) -> str:
    """Deterministic markdown rendering of an episode index record."""

def resolve_concept_candidate(
    candidate_name: str,
    catalog: dict[str, list[dict]],
) -> str | None:
    """Best-effort resolve a candidate concept name to an existing wiki slug.
    Resolution order:
      1. Exact case-insensitive match on article title → unique match wins.
      2. Exact case-insensitive match on any entry in the article's `aliases` → unique match wins.
      3. (NO tag-based resolution by default — tags are too broad for dedup signal.)
    If step 1 or step 2 produces multiple matches across the catalog, return None
    (ambiguous — Haiku will canonicalize at extraction time with full context).
    Returns slug on unique match, None on no-match or ambiguous-match."""
```

### 8. Backfill command

**Command:** `/kb-publish backfill-index [--episode N | --all]` (new subcommand)

**Algorithm, per episode:**

1. Read episode entry from `episodes.yaml`. Skip unless `status: published`.
2. Resolve audio path: the `audio` field is a basename (e.g. `podcast-hardware-2026-04-12.mp3`). Resolve to an absolute path by checking (in order) `<notebooklm.output_path>/<audio>` then `<notebooklm.output_path>/notebooklm/<audio>`. If not found, warn and skip this episode.
3. Check for existing transcript:
   - If the registry has `transcript.markdown` set AND that file exists: use it as-is; skip to step 4.
   - Else look for `<audio_dir>/<audio_stem>.transcript.md` on disk (the naming convention the new feature writes). If found, use it and patch the registry.
   - Otherwise: invoke `transcribe_audio.py` with `--vtt-offset-seconds 0`. Rationale: pre-feature episodes have no intro-music assembly, so the delivered audio IS the raw transcribed audio — offset 0 is correct. The backfilled VTT aligns to the distributed MP3 (which listeners may already have), not to any reassembled version. After transcription, patch the registry: set `transcript.applied = true`, `transcript.vtt = <basename>`, `transcript.markdown = <basename>`, `transcript.speaker_count = <from JSON>`.
4. Run the full extraction pipeline (SAME as step 8c steps 2–8):
   a. Read transcript.
   b. Build catalog + recent-episodes context.
   c. Call Haiku with the extraction prompt.
   d. Validate JSON response. On failure, log and skip this episode, move to next.
   e. Recompute `existed_before` from filesystem.
   f. Build `coverage_map` from all episodes EXCLUDING this episode's ep_id, then call `compute_depth_deltas()`.
   g. Call `index_episode_transactional(...)`.
5. Update `episodes.yaml` for this episode per step 8c's step 9.
6. Report progress per section 8's progress line format.

The extraction-to-transactional orchestration is the same in step 8c and in backfill; both call sites use the same `orchestrate_episode_index()` helper (defined in section 7) rather than duplicating the pipeline. See section 7's API.

**Explicit transcription invocation** (step 3 of the per-episode loop, when a new transcript must be produced):

```bash
source <notebooklm_venv>/bin/activate && \
  python3 <notebooklm_skill_dir>/scripts/transcribe_audio.py \
    --audio <absolute_audio_path> \
    --hosts '<json-encoded host_pool from integrations.notebooklm.podcast.hosts + extra_host_names>' \
    --output-vtt <audio_dir>/<audio_stem>.vtt \
    --output-md <audio_dir>/<audio_stem>.transcript.md \
    --vtt-offset-seconds 0 \
    --model <integrations.notebooklm.podcast.transcript.model or 'large-v3'> \
    --device <integrations.notebooklm.podcast.transcript.device or 'auto'> \
    --language <integrations.notebooklm.podcast.transcript.language or 'zh'> \
    --title "全栈AI — <episode.topic> (<episode.date>)" \
    --json
```

Parse the JSON from stdout. On success, update the registry's `transcript.*` fields as described in step 3 above. On failure (any non-zero exit), skip this episode and continue to the next.

`<notebooklm_skill_dir>` and `<notebooklm_venv>` are resolved from `integrations.notebooklm.venv_path` and the skill install path; kb-publish does NOT duplicate the dependency install.

**Progress reporting:** Print a line per episode: `EP1: transcribing... done (3:42). Extracting... 22 concepts, 4 stubs. Written wiki/episodes/ep-01-gpu-computing-cuda.md`.

**`--all` behavior:** Iterate over every published episode in `episodes.yaml` in order. Continue on per-episode errors (log + skip).

**For the current KB:** EP1 (`podcast-hardware-2026-04-12.mp3`) and EP2 (`podcast-attention-2026-04-12.mp3`) both lack transcripts; backfill will transcribe both before indexing.

### 9. Error handling

| Condition | Behavior |
|---|---|
| Transcript missing at publish time | Skip indexing, log "no transcript", episode still published. `backfill-index` can retry later. |
| Haiku returns malformed JSON | Retry once. On second failure, abort indexing for this episode, leave no partial data. Re-runnable via `--reindex`. |
| Haiku returns concept with unknown `depth_delta_vs_past` value | Treat as validation failure, abort indexing cleanly. |
| Auto-stub collision (existing non-stub at slug) | Do nothing. Log "existing article at `<slug>`; Haiku's proposal ignored." |
| Auto-stub collision (existing stub from another episode) | `created_by` is IMMUTABLE. Update only `last_seen_by` (always), `best_depth_episode` + `best_depth` (when the new episode's depth exceeds the stored best), and append `referenced_by`. Prose sections untouched. |
| `wiki/episodes/ep-<id>-*.md` write fails (disk full, permissions) | Log, abort this episode's indexing, leave `episodes.yaml` untouched (registry not updated). Publish still counts as successful. |
| Dedup judge (step 5b) finds a published episode without a `wiki/episodes/` article | Surface to user: "EPN published but not indexed — run `backfill-index --episode N`." Proceed with what we have. |
| Concept catalog empty (fresh KB) | Haiku gets empty catalog context; every concept becomes `existed_before: false` and gets auto-stubbed. First episode seeds the catalog. |

## Scope, non-goals, and open questions

**In scope:**
- Two new files (`episode_wiki.py`, `test_episode_wiki.py`) and two new prompts (`episode-wiki-extract.md`, `dedup-judge.md`).
- Edits to `kb-publish/SKILL.md` (add step 8c + backfill command) and `kb-notebooklm/SKILL.md` (replace step 5b).
- Backfill for EP1 + EP2 with real transcription.

**Out of scope:**
- Retroactive rewriting of existing wiki articles.
- Changes to the podcast generation or transcription pipeline.
- Multi-language concept merging beyond what Haiku's canonicalization does.
- Any SQLite, FTS, or vector-store infrastructure.

**Open questions resolved during peer review:**

- `wiki_dir` is the existing `integrations.notebooklm.wiki_path` value. Slugs always include the `wiki/` prefix; `slug_to_wiki_relative_path()` strips the prefix when computing paths relative to `wiki_dir`, producing e.g. `<vault>/wiki/quantization/k-quants.md` from slug `wiki/quantization/k-quants`.
- Auto-stubs ARE in the canonicalization catalog (flagged as weak), so future episodes canonicalize to them. This prevents duplicate stubs.
- `scan_episode_wiki` has strict/lenient modes: lenient for dedup reads (skip-with-warning), strict for the reindex staging smoke-parse.
- `kb-publish backfill-index` reuses the existing notebooklm venv (already has `faster-whisper` + `pyannote` installed via the intro-music feature's kb-init setup). kb-publish does NOT gain its own transcription dependency; it shells out to the kb-notebooklm script just like the normal publish flow does for post-processing.
- **Registry schema migration for `concepts_covered[]`:** The current schema supports both `list[str]` and `list[{name, depth}]` forms (existing entries use the object form per kb-publish/SKILL.md §Episode Registry). Indexing step 9 writes the object form. Any downstream readers must handle both; this is a pre-existing schema tolerance, not a new concern.
- **Prior-episode ordering for tie-breaking:** lowest `episode_id` wins. Deterministic; matches publish chronology.
